# 模版：故障/问题总结（summary_problem）

本模版用于 Organizer 插件预处理：当 50_Raw 中的条目被识别为「故障/问题」类时，按以下结构从原文抽取并填充字段，再继续分类归位。

## 目标结构

AI 从原文中抽取信息，按以下格式整理（可选用 YAML frontmatter + Markdown 正文，或纯 Markdown 小标题）。

### 字段说明

| 字段 | 说明 | 可留空 |
|------|------|--------|
| 提出人 | 反馈/上报问题的人或团队 | 是，缺失时填「待补」 |
| 时间点 | 问题发生或上报的时间 | 是，缺失时填「待补」 |
| 现象 | 故障现象、报错信息、用户描述 | 否，尽量从原文提取 |
| 原因/根因 | 已知或分析得出的原因 | 是，待分析可填「待分析」 |
| 处理/方案 | 已采取或建议的处理措施 | 是，待处理可填「待处理」 |
| 状态 | 如：待分析、处理中、已关闭 | 是，默认「待处理」 |

### 输出格式示例（Markdown）

```markdown
---
type: summary_problem
提出人: 
时间点: 
现象: 
原因: 
处理: 
状态: 待处理
---

# 故障/问题摘要

## 现象
（从原文抽取或概括）

## 原因/根因
（从原文抽取，无则留空或写「待分析」）

## 处理/方案
（从原文抽取，无则留空或写「待处理」）

## 原始内容
（可选：保留一段原文便于追溯）
```

## 抽取指引

- **提出人**：原文中的「谁反馈」「谁报」「联系人」「上报人」等。
- **时间点**：原文中的日期、时间、「何时发生」「何时发现」等。
- **现象**：报错信息、异常表现、用户描述的现象、日志片段等，尽量保留关键句。
- **原因/根因**：若原文有分析或结论则抽取；无则填「待分析」。
- **处理/方案**：已采取的措施、临时方案、建议方案等；无则填「待处理」。
- 若某字段原文中完全无信息，填「待补」或「待分析」/「待处理」，不编造内容。
- 整理后保留与原文一致的语义，不改变事实性描述。
